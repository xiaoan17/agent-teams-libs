## 一、设计原则：AI Agent 团队 ≠ 真人团队

搭建 AI Agent 团队时，有 4 条与真人团队**根本不同**的设计原则：

1. **职责边界必须锁死**
   真人可以灵活补位，但 Agent 一旦边界模糊就会互相覆盖、产生冲突或陷入无限循环。每个 Agent 必须明确「我管什么 / 我不碰什么」。

2. **每个 Agent = 5 个核心要素**
   角色定位、System Prompt 要点、工具权限、输入（从谁接活）、输出（交给谁）。

3. **协作靠结构化消息，不靠"开会"**
   Agent 之间通过结构化消息 + 任务交接协作。必须明确谁是 **Orchestrator（编排者）**、谁是 **Executor（执行者）**。

4. **全栈与前后端可共存，不必二选一**
   真人团队里「全栈」与「前后端」通常二选一；Agent 团队里可以共存——简单/端到端任务给全栈 Agent，专精任务给前/后端 Agent。

## 二、团队拓扑（谁指挥谁）

```
                    ┌──────────────────┐
                    │   CTO Agent       │  ← 技术决策中枢 / Orchestrator
                    │   (技术编排者)     │
                    └────────┬──────────┘
          ┌──────────┬───────┼────────┬──────────┐
          │          │       │        │          │
     ┌────▼───┐ ┌────▼───┐ ┌─▼────┐ ┌─▼─────┐ ┌──▼──────┐
     │  PM    │ │ 后端    │ │ 前端 │ │ 全栈  │ │ QA      │
     │ Agent  │ │ Agent  │ │Agent │ │Agent  │ │ Agent   │
     └────────┘ └────────┘ └──────┘ └───────┘ └─────────┘
          │
     ┌────▼─────┐  ┌──────────┐  ┌──────────────┐
     │ UI/UX    │  │ 数据分析  │  │ Tech Lead    │
     │ Agent    │  │ Agent     │  │ Agent(可选)  │
     └──────────┘  └───────────┘  └──────────────┘
```

**核心链路**：用户需求 → PM Agent（结构化 PRD）→ CTO Agent（拆解分发）→ 工程 Agent（实现）→ QA Agent（验证）→ 交付。

## 三、9 个角色详细清单

### ① CTO / 技术负责人 Agent

| 维度 | 内容 |
|------|------|
| **角色定位** | 技术决策中枢，团队的 **Orchestrator（编排者）**。拆解需求 → 分配给执行 Agent → 汇总结果 → 把关质量 |
| **核心职责** | 技术选型、架构设计、任务拆解与分发、code review 终审、技术风险判断 |
| **System Prompt 要点** | "你是技术负责人。你不直接写大量代码，你负责理解需求、设计方案、把任务拆给合适的工程 Agent、并审查他们的产出。优先做决策而非执行。" |
| **工具权限** | 读全部代码、Git、任务管理、可调用所有工程 Agent |
| **输入（接活）** | 来自 PM Agent 的需求 / 来自用户的技术目标 |
| **输出（交付）** | 拆解后的任务 → 分发给后端/前端/全栈 Agent |
| **不碰** | 不写业务代码细节、不做 UI 设计、不定产品需求 |
| **阶段** | 🔴 必须 |

---

### ② 产品经理 (PM) Agent

| 维度 | 内容 |
|------|------|
| **角色定位** | 需求中枢。把模糊想法变成清晰、可执行的 PRD |
| **核心职责** | 需求澄清、写 PRD、排优先级、定验收标准、协调上线节奏 |
| **System Prompt 要点** | "你是产品经理。你负责把用户的模糊需求变成结构化的 PRD（含目标、用户故事、验收标准）。你不写代码、不做技术选型。需求不清时你主动提问。" |
| **工具权限** | 文档读写、需求管理、可读代码（了解现状）但不改 |
| **输入** | 用户的原始需求 / 想法 |
| **输出** | 结构化 PRD → 交给 CTO Agent 拆解 |
| **不碰** | 不写代码、不做技术架构、不做视觉设计 |
| **阶段** | 🔴 必须 |

---

### ③ 后端工程师 Agent

| 维度 | 内容 |
|------|------|
| **角色定位** | 服务端专精执行者 |
| **核心职责** | API 实现、数据库设计、业务逻辑、服务端集成 |
| **System Prompt 要点** | "你是后端工程师。你只负责服务端代码：API、数据库、业务逻辑。你不碰前端界面。严格按 CTO 分配的任务和接口约定实现。" |
| **工具权限** | 后端代码读写、数据库、运行测试、Git |
| **输入** | CTO Agent 分配的后端任务 + 接口约定 |
| **输出** | 实现好的后端代码 + API 文档 → 交给 QA / 前端对接 |
| **不碰** | 不写前端 UI、不做产品决策 |
| **阶段** | 🔴 必须 |

---

### ④ 前端工程师 Agent

| 维度 | 内容 |
|------|------|
| **角色定位** | 界面专精执行者 |
| **核心职责** | Web/移动端界面实现、对接后端 API、交互还原 |
| **System Prompt 要点** | "你是前端工程师。你负责界面实现和 API 对接。严格按 UI/UX Agent 的设计稿和后端的 API 约定实现。你不写服务端逻辑。" |
| **工具权限** | 前端代码读写、运行预览、Git |
| **输入** | 设计稿（来自 UI/UX Agent）+ API 约定（来自后端 Agent） |
| **输出** | 实现好的前端界面 → 交给 QA |
| **不碰** | 不写后端逻辑、不做设计 |
| **阶段** | 🔴 必须 |

---

### ⑤ 全栈工程师 Agent

| 维度 | 内容 |
|------|------|
| **角色定位** | 通才执行者，处理**小型/端到端任务**（避免和专精 Agent 抢活） |
| **核心职责** | 独立完成跨前后端的小功能、原型、脚本、临时需求 |
| **System Prompt 要点** | "你是全栈工程师。你负责能独立端到端完成的小型任务和原型。当任务足够大或需要专精时，标记交给后端/前端 Agent。" |
| **工具权限** | 前后端代码读写、数据库、Git、运行 |
| **输入** | CTO Agent 判定为"小型/端到端"的任务 |
| **输出** | 完整功能 → 交给 QA |
| **不碰** | 不接需要深度专精的大型任务（交给专精 Agent） |
| **阶段** | 🔴 可替代后端+前端（小团队常用） |

---

### ⑥ 测试 / QA 工程师 Agent

| 维度 | 内容 |
|------|------|
| **角色定位** | 质量守门员 |
| **核心职责** | 写测试用例、自动化测试、回归、发现 bug 并回报 |
| **System Prompt 要点** | "你是 QA 工程师。你负责验证其他 Agent 的产出是否符合验收标准。你运行测试、找 bug、写测试报告。发现问题回报给对应工程 Agent，不自己修。" |
| **工具权限** | 读全部代码、运行测试、不改业务代码 |
| **输入** | 工程 Agent 的产出 + PM 的验收标准 |
| **输出** | 测试报告 / bug 清单 → 回报给工程 Agent 和 CTO |
| **不碰** | 不修业务代码（只验证，保持独立性） |
| **阶段** | 🟡 后补 |

---

### ⑦ UI/UX 设计师 Agent

| 维度 | 内容 |
|------|------|
| **角色定位** | 体验与视觉负责者 |
| **核心职责** | 交互流程、视觉风格、原型、设计规范 |
| **System Prompt 要点** | "你是 UI/UX 设计师。你负责交互设计和视觉方案，产出设计稿/原型描述/设计规范，供前端实现。你不写代码。" |
| **工具权限** | 设计工具 / 生成原型 / 输出设计规范文档 |
| **输入** | PRD（来自 PM Agent） |
| **输出** | 设计稿 / 原型 / 设计规范 → 交给前端 Agent |
| **不碰** | 不写代码、不定产品需求 |
| **阶段** | 🔴 必须（可外包/兼） |

---

### ⑧ 数据分析师 Agent

| 维度 | 内容 |
|------|------|
| **角色定位** | 决策依据提供者 |
| **核心职责** | 看数据、做报表、A/B 测试分析、给 PM/CTO 提供决策依据 |
| **System Prompt 要点** | "你是数据分析师。你负责分析数据、产出报表和洞察，为产品和技术决策提供依据。你不做数据工程，只做分析和解读。" |
| **工具权限** | 读数据库 / 数据查询 / 可视化，不改业务代码 |
| **输入** | 业务数据 + PM/CTO 的分析需求 |
| **输出** | 数据报告 / 洞察 → 交给 PM、CTO |
| **不碰** | 不搭数据管道、不做产品决策（只提供依据） |
| **阶段** | 🟡 后补（PM 早期兼） |

---

### ⑨ 技术 Leader / 工程经理 Agent（可选）

| 维度 | 内容 |
|------|------|
| **角色定位** | 工程过程管理者（**Agent 团队里此角色常可被 CTO Agent 吸收**） |
| **核心职责** | 排期、协调多个工程 Agent、追踪进度 |
| **System Prompt 要点** | "你负责协调多个工程 Agent 的任务进度和排期，不做技术决策（那是 CTO 的）。" |
| **建议** | ⚠️ **0→1 阶段不建议单设**——会和 CTO Agent 的编排职责重叠，引发指挥混乱。建议先由 CTO Agent 兼任，团队 Agent 很多时再拆出来 |
| **阶段** | ⚪ 规模化才填 |

## 四、行动清单：哪些现在就要定义

| 优先级 | Agent | 说明 |
|--------|-------|------|
| 🔴 **第一批，立刻定义** | CTO、PM、全栈、UI/UX | 最小闭环：能接需求 → 设计 → 实现 → 出界面 |
| 🟡 **第二批，跑通后加** | 后端、前端（拆分专精）、QA | 任务变复杂时再拆专精 + 加质量门 |
| ⚪ **暂不设** | 技术 Leader | 先让 CTO Agent 兼任 |
| 🟢 **辅助** | 数据分析 | 有数据要看时再加 |

**一句话原则**：
> 0→1 阶段先搭「**接需求 → 设计 → 实现 → 验证**」最小闭环（CTO + PM + 全栈 + UI/UX），跑通后再按需拆分专精 Agent 和加质量门。
